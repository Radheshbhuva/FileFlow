# AWS Cognito Migration Guide

This document describes the design, implementation, environment variables, and synchronization steps required to migrate FileFlow authentication to Amazon Cognito.

---

## 1. Environment Configurations

To migrate authentication to AWS Cognito, configure the environment variables below in the active backend settings:

```env
# Active authentication driver selection: "cognito" or "mock"
AUTH_PROVIDER=cognito

# AWS Region & Cognito Credentials
AWS_REGION=us-east-1
COGNITO_USER_POOL_ID=us-east-1_xxxxxxxxx
COGNITO_CLIENT_ID=yyyyyyyyyyyyyyyyyyyyyyyyyy
COGNITO_CLIENT_SECRET=zzzzzzzzzzzzzzzzzzzzzzzzzzzzzzzzzzzzzzzzzzzz
COGNITO_IDENTITY_POOL_ID=us-east-1:aaaa-bbbb-cccc-dddd-eeee
```

---

## 2. Directory Schema & Attributes

The Cognito User Pool maps and enforces the following user profile attributes:
*   **`email`**: User email address (Username).
*   **`name`**: User's full name.
*   **`sub`**: Core subject UUID generated by Cognito.
*   **`email_verified`**: Mapped verification flag.
*   **`custom:role`**: Workspace role (`OWNER`, `ADMIN`, `MANAGER`, etc.).
*   **`custom:planType`**: Tenant subscription tier (`FREE`, `PRO`, `ENTERPRISE`).
*   **`custom:workspaceId`**: Primary workspace identifier.

---

## 3. Database Schema Enhancements

To link external Cognito subjects back to internal DynamoDB records, a new indexing attribute has been added to the User entity:

*   **`cognitoSub`**: Represents the unique AWS Cognito subject UUID mapping.
*   **DynamoDB GSI Mappings**: Queries looking up users will query GSI1 (`USER_EMAIL#<email>`) or search directly via the primary partition `USER#<userId>` containing the mapped profile.

---

## 4. User Synchronization (`UserSyncService`)

The synchronization layer is responsible for aligning external Cognito claims with the local DynamoDB database profile:

1.  **Write-Through Signup**: During registration, Cognito creates the credentials, and the sync service automatically writes the corresponding local User record, provisioning a default workspace.
2.  **On-Demand Migration**: If a legacy user logs in, Cognito authenticates their email, and `UserSyncService` links their profile by attaching the `cognitoSub` to the existing DynamoDB profile record, maintaining zero-downtime.

---

## 5. Phased Zero-Downtime Migration Strategy

To ensure zero downtime during authentication switchovers, the system supports a hybrid verification phase:

1.  **Dual Token Validation**:
    The middleware (`protect.ts`) reads incoming Authorization headers and parses token headers.
    *   If the token issuer (`iss`) matches the Cognito User Pool, it validates using the RS256 public key lookup (JWKS).
    *   If the token issuer is local/mock, it falls back to HS256 local verification.
2.  **Gradual Password Migration**:
    Legacy hashes are checked during the migration window. Once a user logs in successfully, the backend updates their credentials in Cognito, setting their password, and flags the database password as `COGNITO_MANAGED`.
